Guidelines:

I want this to be a git enabled project

This is a new project, all source code should be under ./src and all test code should be under ./test

This should be a collection of dockerized containers that interface with eachother and are controlled by a docker-compose, but should be capable of running under kubernetes.

All supporting files should either be at the root or under appropriately named directories (ie. ./docs)

I want comprehensive documentation, both on what the application does and how to use it

I want Well written code that is maintainable and follows best practices

I want Well written and comprehensive unit tests

I want UI based tests

I want Component tests that verify the application as a whole

The language it is written in is not important, but I want it to be as fast as possible, so choose a language using that as the main criteria



Process for Completing this:

Phase 1: Plan the project

Phase 2: Follow through on the plan, be sure to handle unit tests by following TDD

Phase 3: Verify, run unit tests, run component tests, run any UI based tests

Phase 4: Compile supplemental documentation and instructions

Phase 5: Create the release pipeline, including versioning, test, and compile/release process, keep in mind the various platforms to support



Project Details:

This is meant to be an all encompasing build pipeline for applications.  For the purposes of this project it will start as a docker-compose managed system, but the goal is for it to be able to be adapted to run under kubernetes with auto-scaling in mind.

Product shape — have some fun with it: adaptive-pipe is a multi-tenant SaaS. One hosted installation serves many customers; each customer (tenant) gets isolated data, credentials, repos, builds, and settings. No peeking at the neighbor's pipelines. (What you deploy with Terraform—single-tenant app or multi-tenant product on AWS—is still your own business; this rule is about the pipeline product itself.)

The goal is to have a "microservice" node based architecture with the below nodes, each node should be multi-node scalable (except Database as noted):

- UI node.  Always running (if app is available) and displays progress/status/stages/etc.  Should include a platform settings area to configure a "warm pool": number of spooled/idle workers for Build, Test, and Deploy stage nodes (see worker scaling below).

- Orchestrator node. Handles the interfacing and calls between nodes. hosts the external APIs. Keeps track of everything. Always running

- File node.  This is meant to be a "cache" of sorts, at the launch of the pipeline this will download the source files and be the home for any file operations.  Always Running.  MVP: local volume storage; design should allow future cloud object storage (S3/GCS/Azure Blob) behind the same service.

- Build node.  This is meant to be where building and language based testing occurs.  There may need to be multiple dockerfiles to manage different language capabilities.  Workers should be available quickly (warm pool configurable in UI); scale-from-zero is acceptable when pool is zero.  Not long-lived per job.

- Test node.  This should be able to handle whatever non-language based testing occurs, think UI tests, API/Contract tests, Performance tests (will work in conjunction with the deploy node to ensure env is up first).  Same scaling model as Build (warm pool + scale-from-zero).  Not long-lived per job.

- Deploy node.  MVP: deploy to AWS using Terraform.  Architecture and config must make it easy to add Helm, Ansible, GCP, Azure, and smarter auto-detection of IaC/cloud later.  Same scaling model as Build/Test.  Spins up after testing phase completes, then spins down after.

- Validate node.  This is meant to be a "shift left" node, it should validate that the inputs and any build/test/deploy criteria is met and valid prior to beginning even the build step. Should operate off a boolean flag on whether to connect to cloud providers to determine if pre-reqs are met there (ie, is VPC available for a deploy to connect under).  This one should always be running.

- Database.  The only non-multi-node one.  MVP: small containerized PostgreSQL for logs and historical status.  Future: may be swapped for an external or more production-grade database without changing product goals.  Always running for a given deployment.



Supported items.  These should be configured in a way where it's easy to expand to additional languages/IAC/Clouds

- Languages to support (six): Java, Python, TypeScript/JavaScript, Rust, Go, C#

- IaC providers: MVP Terraform.  Future goals: Helm, Ansible, and additional interfaces; avoid hard-coding so new providers are config + worker image extensions.

- Cloud providers: MVP AWS.  Future goals: GCP, Azure; same extension model.



Orchestrator details.

- When the kickoff API is called, it should return immediately with an appropriate HTTP success response (use 200 OK when the kickoff is accepted and recorded; use another 2xx only if a future API shape warrants it).

- Queue: Redis for build/stage work tracking and handoff.

- Inter-service communication: REST (JSON) for internal calls unless a future version explicitly adds another style.

- It should use the queue so that once an appropriate node is ready, work is taken and handed off. Once a node is done and replies to the Orchestrator for that stage, it should appropriately update the UI, and place the "build" on a queue for the next stage.

- Handles all inter-node communication. Nodes should only talk to the Orchestrator or the File node(s)

- Stores all information to the Database. Retrieves "completed" data from the database.

- Inbound API call will:

- - include Github Org & Repo, along with commit information.

- - be able to be initiated and report status back to GitHub CI process

- GitHub MVP: github.com, commit-triggered hook to initiate runs.  Future: GitHub Enterprise Server / on-prem API bases.  How status is surfaced on GitHub (Checks vs Commit Status, etc.) is left to implementation with stable external behavior documented in ./docs.

- Should "maintain" node hold.  If there are 3 file nodes, it should be sticky to the first one it gets, same process applies to the other nodes.

- Should keep material data for the last 10 builds per (Tenant, Github Org, Repo) combination, regardless of build success/failure status, so any of those 10 can be initiated at any stage.  Retention is independent per repo within a tenant (tenant A org1/repo1 has 10, tenant A org1/repo2 has another 10, tenant B has its own counts, etc.).

- Should handle any credentials/secure store in conjunction with the UI.  Will need to encrypt and store them in their own table on the DB. User must be "admin" level to add/update.

- Non-functional expectations (detail in ./docs): threat-aware API and credential handling, structured logging and trace correlation, idempotency for webhooks/kickoff where needed, rate limits on public routes, optional feature flags for deferred capabilities, backup/RPO considerations when moving past dev-only.



UI details.

- Communicates through the orchestrator to get any information.  Does not need to talk to any other node directly.

- Should organize builds by Github Org and Repo.  Then individual builds are numbered sequentially.  If version information and commit information is available, tag it to the build number.

- Should display each stage in-order, if for some reason a stage is skipped, it should still display it, just don't include progress.

- Should display an estimated time remaining for each stage based on historical data.  MVP: simple moving average; implementation should isolate the function so it can be replaced later (exponential weighting, etc.).

- Should allow a user to "kick off" a build of their own.  Will use last commit information to ensure that code is what is expected.

- Should allow a user to kick off a build at a specific stage as long as it has the necessary information/data available.

- Should have a credential page to enter secure strings.

- Should have User login page.  MVP: local username/password.  Design should allow future auth methods (OIDC, SSO, etc.) without rewriting core flows.

- Platform settings: configure warm pool sizes (spooled workers) for Build, Test, and Deploy worker pools (scoped per tenant in the SaaS model).
